## mental-maths — 3 finding(s)
### 1. [MAJOR] Free Play tap targets fall below the 44px minimum at the documented minimum host width (640px)
**Evidence:** js/anims/mental-maths.js lines 121-123:
  const n = L.hi - L.lo + 1;
  const avail = Math.min(900, (host.clientWidth || 640) - 20);
  L.W = Math.max(34, Math.min(64, Math.floor(avail / n)));
ANIM_BRIEFS.md's contract explicitly requires 'Handle host widths 640-980px' (line 29). At host.clientWidth=640 (the documented lower bound), avail = 620.
For the Free Play sandbox mission '61 − 9 = 52 via 60' (SANDBOX[2], line 51): friend=floor(61/10)*10=60, target=52, lo=min(52,60,61)-3=49, hi=max(52,60,61)+3=64, n=16 ticks. L.W = floor(620/16) = 38px.
For '24 + 8 = 32 via 30' (SANDBOX[0], line 49): friend=30, lo=21, hi=35, n=15 ticks. L.W = floor(620/15) = 41px.
Child scenario: on an iPad at the low end of the module's own supported width range, a 10-year-old tapping quickly on Free Play's widest number lines is working with 38-41px-wide tick buttons — under Apple's 44pt touch-target minimum and the review's explicit '<44px' criterion — so an intended tap on the correct tick can easily register on the adjacent number instead, producing an undeserved 'miss' nudge.
**Suggested fix:** Raise the hard floor from 34 to 44 (e.g. Math.max(44, Math.min(64, Math.floor(avail/n)))) and let the wrap scroll horizontally when totalW would exceed the host width, rather than letting individual ticks shrink under the minimum touch-target size across the documented 640-980px range.

### 3. [MINOR] Double-tapping the glowing drop-target column while the carry cannonball is mid-flight starts a second competing tween on the same DOM node, snapping the ball backward and firing a duplicate 'drop' sound

### 1. [MINOR] Resize mid-drag leaves Sir Roundbottom permanently skewed (stray inline transform never cleared)
**Evidence:** In js/anims/rounding.js the drag options are: `onMove(dx) { const cl = Math.max(-16, Math.min(16, dx * 0.4)); const rot = Math.max(-10, Math.min(10, dx * 0.14)); sirEl.style.transform = `translateX(${cl}px) rotate(${rot}deg)`; }, onEnd() { sirEl.style.transform = ''; triggerRoll(); }` (lines ~221-226). ## machines-mystery — 1 finding(s)
### 1. [MAJOR] Resize during a chip-tap tween (not just the RUN-IT forward demo) corrupts mission state into an unrecoverable soft-lock
**Evidence:** In `onChipTap` (js/anims/machines-mystery.js:253-283), tapping the CORRECT chip does: `win.solved = true; win.chosen = chip; buildReverseDOM(); showChips(false);` synchronously (so the window immediately renders as solved/green with the answer showing), THEN starts `animateTokenTo(winIdx, newVal, callback)` — and only inside that tween's completion callback does `activeWin += 1`, `tokenVal = newVal`, and (for non-final windows) `showChips(true); renderChips();` run.

`animateTokenTo` stores its cancel handle in the single shared `cancelTok` (line 173). The debounced resize handler does `if (cancelTok) { cancelTok(); cancelTok = null; }` (line 417) before falling into the `phase === 'solving'` branch (lines 432-437), which just does `buildReverseDOM(); positionTokenInstant(idx, tokenVal); busy = false;` — it does NOT re-run the tween's completion logic.

Critically, `tween()`'s own cancel function (js/anims/_kit.js:141-145) is `() => { settled = true; if (raf) cancelAnimationFrame(raf); if (guard) clearTimeout(guard); }` — it never calls `apply(to)` nor `done()`. So if a resize fires while a correct-chip tween is in flight, cancelling it means `activeWin` never increments, `tokenVal` never updates to the post-step value, and (for non-final windows) `showChips(true)`/`renderChips()` never run to re-populate the chip tray.

Result: the just-solved window permanently shows its answer (green, e.g. '÷ 2'), but `activeWin` still points at that same window, `chipsRow` is empty and hidden (it was set to `showChips(false)` synchronously before the tween started and nothing re-shows it), the lever and RUN-IT button are both hidden in the 'solving' phase, and the token sits at a stale position/value.
